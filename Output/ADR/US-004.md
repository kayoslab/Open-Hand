ADR-004: Responsive Application Shell Layout Strategy\n\nStatus: Accepted\nDate: 2026-04-27\nTicket: US-004\n\nContext:\nThe application needs a responsive shell that works across mobile, tablet, and desktop. No layout components, CSS files, or design tokens exist yet. This is the first UI structural work — choices made here (breakpoints, token system, layout approach) will constrain all future UI development.\n\nDecision:\n- Establish CSS custom properties in src/ui/tokens.css for colors, spacing, typography, and breakpoints (mobile <768px, tablet 768-1024px, desktop >1024px)\n- Use CSS Modules for scoped component styling (supported by Vite out of the box)\n- Create Layout, Header, and Nav components under src/ui/ following existing repository conventions\n- Use semantic HTML regions (header, nav, main) with CSS Grid/Flexbox for responsive layout\n- Navigation links are placeholder anchors — no router installed yet\n- Prevent horizontal scroll via overflow-x constraints\n\nConsequences:\n- Breakpoint values become project-wide conventions — all future responsive work should reference tokens.css\n- CSS Modules pattern is established as the styling approach for the project\n- Existing App.test.tsx will need updating since App.tsx content changes from \"Card Deck v2\" heading to the Layout shell\n- Navigation links must be implemented as plain anchors/buttons that won't break when a router is added in a future ticket\n- No design mockups exist; the aesthetic interpretation of \"warm, premium, calm\" may require future iteration\n\nRisks accepted:\n- Subjective aesthetic without mockups — acceptable for initial shell, can iterate\n- Placeholder navigation without routing — low risk, standard pattern for incremental delivery
